# SPEC-ドラッグドロップ移動: サイドバーカテゴリーツリーのドラッグドロップ移動

> **最終更新**: 2026-04-28  
> **担当者**: —  
> **関連 Issue**: —

## 背景・目的

- **背景**: 現状のカテゴリーツリーでは、カテゴリーやリクエストを別のカテゴリーへ移動する手段がない。整理のためには削除・再作成するしかなく、コレクションが増えるほど管理が煩雑になる。
- **目的**: カテゴリーおよびリクエストをドラッグドロップで直感的に移動できるようにし、コレクション管理の利便性を高める。
- **スコープ（含むもの）**: カテゴリーツリー内でのカテゴリー・リクエストの移動操作。
- **スコープ（含まないもの）**: ツリー内でのアイテムの並び替え（ソート順の変更）、履歴パネルへのドロップ、他ウィンドウ・タブ間の移動。

---

## 機能要件

### 必須要件（Must Have）

- [ ] カテゴリーをドラッグして別のカテゴリーへ移動できる
- [ ] リクエストをドラッグして別のカテゴリーへ移動できる
- [ ] リクエストをドラッグしてルートレベル（未分類）へ移動できる
- [ ] カテゴリーをドラッグしてルートレベルへ移動できる
- [ ] ドラッグ開始条件は **1 秒間の長押し（ロングプレス）** とし、通常のクリック・ダブルクリックと区別する
- [ ] ドラッグ中はドラッグ対象アイテムを半透明（opacity 50% 程度）で表示する
- [ ] ドラッグ中に有効なドロップ先をハイライト表示する
- [ ] カテゴリー上にドロップした場合、そのカテゴリーの子として移動する
- [ ] ルートエリアにドロップした場合、ルートレベル（`parentId: null` / `categoryId: null`）に移動する
- [ ] カテゴリーをその子孫カテゴリーへ移動しようとした場合は無効とし、ドロップを拒否する（循環参照防止）
- [ ] Escape キー押下でドラッグをキャンセルできる
- [ ] 無効なドロップ先でリリースした場合はキャンセルし、元の位置を維持する
- [ ] カテゴリーを移動した場合、`parentId` のみを更新し、配下のサブカテゴリーとリクエストの構造はそのまま維持する
- [ ] リクエストを移動した場合、`categoryId` のみを更新する

### 推奨要件（Should Have）

- [ ] ドラッグ中にカテゴリーの上でホバーし続けると、そのカテゴリーが自動展開される（展開待機時間: 700ms 程度）
- [ ] ドラッグ中のカーソルを `grabbing` に変更する
- [ ] 無効なドロップ先では禁止カーソル（`not-allowed`）を表示する
- [ ] ドロップ成功後、移動先のカテゴリーを自動展開して移動結果を確認しやすくする

### 将来対応（Nice to Have）

- [ ] ドラッグドロップによるツリー内の並び替え（ソート順の手動変更）
- [ ] タッチデバイス（スマートフォン・タブレット）でのタッチドラッグ対応

---

## 非機能要件

### パフォーマンス

- ドラッグ中のハイライト更新は 16ms 以内（60fps）でレンダリングされること
- カテゴリー・リクエスト数が各 200 件以上の状態でもドラッグ操作が滑らかに動作すること

### セキュリティ

- ドラッグドロップはローカルの IndexedDB データのみを対象とし、外部データの注入は行わない

### アクセシビリティ

- ドラッグドロップに頼らず、コンテキストメニューや既存の移動操作（将来的な「移動先を選択」ダイアログ等）でも同等の操作が可能であることが望ましい（本仕様のスコープ外）
- ロングプレス中は視覚的フィードバック（インジケーター等）を表示し、操作中であることをユーザーに伝える

### ブラウザ対応

- Chrome 最新版・Firefox 最新版・Safari 最新版で動作すること
- HTML5 Drag and Drop API またはポインターイベント（`pointerdown` / `pointermove` / `pointerup`）を使用して実装する
  - HTML5 D&D API はモバイルタッチ非対応のため、ポインターイベントベースの実装を推奨する

## UI/UX 設計

### 画面レイアウト（ドラッグ中の状態）

```
┌─────────────────────────────┐
│  [+新しいカテゴリー] [+新しいリクエスト] │  ← ツールバー
├─────────────────────────────┤
│ ▼ 📂 API Collection         │
│   ▼ 📂 Users ░░░░░░░░░░░░  │  ← ドラッグ中（半透明）
│     📄 GET  Get User        │
│   ▶ 📂 Orders ████████████  │  ← ドロップ先ハイライト（インジゴ枠線）
│     📄 POST Create Order    │
│ ▶ 📂 Auth ░░░░░░░░░░░░░░░  │
│   📄 GET  Health Check      │
├─────────────────────────────┤
│ ──── ルートへドロップ ──────  │  ← ルートドロップゾーン（ドラッグ中のみ表示）
└─────────────────────────────┘
```

### 操作フロー

#### カテゴリー移動

1. ユーザーがカテゴリー行を 1 秒間長押しする
2. システムがドラッグモードを開始し、対象行を半透明表示にする
3. ユーザーがカーソルを移動させると、有効なドロップ先候補（他のカテゴリー行）がハイライトされる
4. 移動先カテゴリーの上でドロップする
   - 有効な場合: システムが `category.parentId` を更新し、ツリーを再描画する
   - 無効な場合（子孫へのドロップ等）: キャンセルされ元の位置を維持する
5. ドロップ成功後、移動先カテゴリーが展開され、移動したカテゴリーが表示される

#### リクエスト移動

1. ユーザーがリクエスト行を 1 秒間長押しする
2. システムがドラッグモードを開始し、対象行を半透明表示にする
3. ユーザーがカーソルをカテゴリー行またはルートドロップゾーンへ移動させる
4. ドロップする
   - カテゴリー上: `request.categoryId` をそのカテゴリーの ID に更新する
   - ルートゾーン: `request.categoryId` を `null` に更新する
5. ドロップ成功後、移動先が展開され移動したリクエストが表示される

#### キャンセル操作

- Escape キー押下: 即座にドラッグをキャンセルし、ビジュアルフィードバックを解除する
- 無効なドロップ先（禁止ゾーン）でのポインターリリース: キャンセル扱いとして元の状態を維持する
- ドラッグ元と同じ位置へのドロップ: 変更なし（キャンセルと同等）

### 表示仕様

| 状態 | 表示 |
|------|------|
| ロングプレス待機中（0〜1 秒） | 対象行の背景色をわずかに変化させてフィードバック（`bg-indigo-500/5` 等） |
| ドラッグ中・ドラッグ対象 | `opacity-50` で半透明表示 |
| ドラッグ中・有効ドロップ先 | `border border-indigo-500` のハイライト枠線 |
| ドラッグ中・無効ドロップ先 | ハイライトなし、カーソルを `not-allowed` に変更 |
| ルートドロップゾーン | ツリー下部に常時表示（ドラッグ中のみ `border-dashed border-indigo-500/40` でアクティブ表示） |

## データモデル変更

### 変更なし

移動操作は既存フィールドの更新のみで実現できる。

- カテゴリー移動: `Category.parentId` を新しい親カテゴリーの ID または `null` に更新
- リクエスト移動: `SavedRequest.categoryId` を新しいカテゴリーの ID または `null` に更新

```typescript
// カテゴリー移動: updateCategory() を使用
await updateCategory(categoryId, { parentId: newParentId });

// リクエスト移動: updateSavedRequest() を使用
await updateSavedRequest(requestId, { categoryId: newCategoryId });
```

いずれも `lib/storage.ts` の既存関数（`updateCategory`, `updateSavedRequest`）で対応可能。スキーマ変更・DB バージョンアップは不要。

## 実装方針

### 推奨アプローチ: ポインターイベントベース

HTML5 Drag and Drop API はタッチデバイス非対応・スタイル制御が困難なため、ポインターイベントを使用したカスタム実装を推奨する。

```
pointerdown → 1秒タイマー開始 → タイマー満了でドラッグモード開始
  ↓（タイマー中にpointermove/pointerupが発生したらキャンセル）
pointermove → ドロップ先判定・ハイライト更新
  ↓
pointerup → ドロップ先が有効ならストレージ更新・ツリー再描画
```

### 状態管理

ドラッグ状態は `CategoryTree` コンポーネント内の `useState` で管理する（Zustand 等の外部ライブラリは使用しない）。

```typescript
type DragState = {
  type: 'category' | 'request';
  id: string;
} | null;

type DropTarget = {
  type: 'category';
  id: string;
} | {
  type: 'root';
} | null;
```

### 循環参照チェック

カテゴリーをドロップ先カテゴリー上にホバーした際、ドロップ先がドラッグ中カテゴリーの子孫かどうかをリアルタイムでチェックする。

```typescript
function isDescendant(candidateId: string, ancestorId: string, categories: Category[]): boolean {
  // candidateId が ancestorId の子孫である場合 true
}
```

`lib/inheritance.ts` の `buildCategoryChain` を参考に実装する。

### コールバック追加（`CategoryTreeProps`）

```typescript
interface CategoryTreeProps {
  // 既存のプロパティ ...
  onMoveCategory: (categoryId: string, newParentId: string | null) => void;
  // onMoveRequest は既に存在する
}
```

`ApiTester.tsx` 側で `onMoveCategory` を実装し、`updateCategory` を呼び出す。

## 受け入れ条件

以下をすべて満たすことでリリース可能とする。

- [ ] AC1: リクエスト行を 1 秒間長押しするとドラッグモードが開始され、別のカテゴリーへドロップするとリクエストが移動すること
- [ ] AC2: カテゴリー行を 1 秒間長押しするとドラッグモードが開始され、別のカテゴリーへドロップするとカテゴリー（および配下のサブカテゴリー・リクエスト）が移動すること
- [ ] AC3: カテゴリーをルートドロップゾーンへドロップすると `parentId: null` のルートレベルカテゴリーになること
- [ ] AC4: リクエストをルートドロップゾーンへドロップすると `categoryId: null` の未分類リクエストになること
- [ ] AC5: カテゴリーをその直接の子カテゴリーへドロップしようとした場合、ドロップが拒否されてカテゴリーが元の位置に留まること
- [ ] AC6: カテゴリーを 2 段階以上深い子孫カテゴリーへドロップしようとした場合も同様に拒否されること
- [ ] AC7: ドラッグ中に Escape キーを押すとドラッグがキャンセルされ、アイテムが元の位置を維持すること
- [ ] AC8: 無効なドロップ先でポインターをリリースした場合、アイテムが元の位置を維持すること
- [ ] AC9: ドラッグ中はドラッグ対象行が半透明になること
- [ ] AC10: ドラッグ中に有効なドロップ先カテゴリーがハイライトされること
- [ ] AC11: 1 秒未満のクリック・ダブルクリックは従来通り（選択・リネーム開始）の動作をすること
- [ ] AC12: 移動後にページをリロードしても移動結果が IndexedDB に永続化されていること
- [ ] AC13: 既存機能（名前変更・削除・複製・リクエスト送信）がドラッグドロップ実装後も正常に動作すること（リグレッションなし）
