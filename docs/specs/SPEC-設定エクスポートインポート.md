# SPEC-設定エクスポートインポート: Settings Export / Import

> **ステータス**: 草稿  
> **作成日**: 2026-04-28  
> **最終更新**: 2026-04-28

---

## 1. 背景・目的

- **背景**: api-tester の全データはブラウザの IndexedDB に保存されるため、ブラウザをまたいだデータ共有・バックアップ・復元の手段がない。別のマシンや別のブラウザで同じ設定を再現するには手作業で再入力するしかない。
- **目的**: カテゴリー（継承設定・変数含む）と保存済みリクエストを JSON ファイルとしてエクスポートし、別環境へインポートできるようにする。
- **スコープ（含むもの）**:
  - `Category[]` と `SavedRequest[]` の JSON エクスポート（ブラウザダウンロード）
  - JSON ファイルのインポート（既存データの全上書き）
  - `lib/storage.ts` への `exportData()` / `importData()` 関数の追加
  - ヘッダーツールバーへの Export / Import ボタンの追加
  - インポート前の確認ダイアログ
  - スキーマバリデーション（構造不正ファイルの拒否）
- **スコープ（含まないもの）**:
  - 履歴（`HistoryItem`）のエクスポート / インポート
  - 既存データとのマージ（差分インポート）
  - 暗号化・パスワード保護
  - クラウドストレージへの保存・読み込み
  - 他ツール（Postman / Insomnia 等）形式との相互変換

---

## 2. 機能要件

### 2.1 必須要件（Must Have）

- [ ] ヘッダーにエクスポートボタン（`Download` アイコン）を配置し、クリックで JSON ファイルをダウンロードできること
- [ ] エクスポートする JSON には `version`・`exportedAt`・`categories`・`requests` フィールドを含めること
- [ ] エクスポートファイル名は `api-tester-export-YYYYMMDD-HHmmss.json` 形式とすること
- [ ] ヘッダーにインポートボタン（`Upload` アイコン）を配置し、クリックでファイル選択ダイアログを開けること
- [ ] ファイル選択ダイアログは `.json` ファイルのみ受け付けること（`accept=".json"`）
- [ ] インポート前に「既存データがすべて上書きされます。続けますか？」という確認ダイアログを表示すること
- [ ] 確認後、既存の全カテゴリーと全保存済みリクエストを削除し、インポートデータに置き換えること
- [ ] インポート完了後、UI（カテゴリーツリー・コレクション一覧）を即座に再フェッチして反映すること
- [ ] インポートするデータのスキーマバリデーションを行い、不正な場合はエラーメッセージを表示してインポートを中断すること

### 2.2 推奨要件（Should Have）

- [ ] エクスポート中・インポート中はボタンを無効化し、二重実行を防ぐこと
- [ ] インポート成功時に「インポートしました（カテゴリー N 件、リクエスト M 件）」というフィードバックを表示すること（トーストまたはヘッダー付近の一時メッセージ）
- [ ] インポートファイルのサイズが 10 MB を超える場合は警告を表示すること（処理は続行可）

### 2.3 将来対応（Nice to Have）

- [ ] 既存データとのマージ（ID 衝突時は上書き or スキップを選択）
- [ ] インポート前のプレビュー（含まれるカテゴリー名・リクエスト数を表示）
- [ ] エクスポート対象の選択（特定カテゴリーのみなど）

---

## 3. 非機能要件

- **パフォーマンス**: カテゴリー 100 件・リクエスト 500 件のデータが 2 秒以内にエクスポート / インポートできること
- **セキュリティ**:
  - インポートは `JSON.parse()` のみを使用し、`eval()` 等の動的コード実行は行わないこと
  - インポート時に既知フィールド以外の値は無視し、未知フィールドを IndexedDB へ書き込まないこと（ストリップ）
  - ファイルサイズ上限は実装上設けないが、10 MB 超は警告を出してユーザーに判断させること
- **アクセシビリティ**: Export / Import ボタンに `title` 属性を付与し、スクリーンリーダーで認識できること
- **ブラウザ対応**: Chrome 最新版・Firefox 最新版・Safari 最新版

## 4. UI/UX 設計

### 4.1 ヘッダーボタン配置

現状のヘッダー（右端）:

```
[テーマ切り替え]
```

Export / Import ボタンをテーマ切り替えボタンの左に追加する:

```
[Import] [Export] [テーマ切り替え]
```

- **Export ボタン**
  - アイコン: `lucide-react` の `Download`（size=16）
  - `title` 属性: `"設定をエクスポート"`
  - スタイル: 既存のテーマ切り替えボタンと統一（`p-2 rounded-lg text-slate-500 hover:text-slate-300 hover:bg-slate-800/60`）
- **Import ボタン**
  - アイコン: `lucide-react` の `Upload`（size=16）
  - `title` 属性: `"設定をインポート"`
  - スタイル: Export ボタンと同一
  - 実装: `<input type="file" accept=".json">` を hidden で配置し、ボタンクリックで `.click()` を呼び出す

### 4.2 ユーザー操作フロー — エクスポート

```
1. ユーザーがヘッダーの [Export] ボタンをクリックする
2. システムが IndexedDB から全カテゴリーと全保存済みリクエストを取得する
3. システムが JSON 文字列を生成し、Blob オブジェクトを作成する
4. システムが <a download="api-tester-export-YYYYMMDD-HHmmss.json"> をプログラム的にクリックする
5. ブラウザのファイルダウンロードが開始される
```

### 4.3 ユーザー操作フロー — インポート

```
1. ユーザーがヘッダーの [Import] ボタンをクリックする
2. ブラウザのファイル選択ダイアログが開く（.json のみ表示）
3. ユーザーがファイルを選択する
4. システムがファイルを読み込み、JSON をパースする
   - パース失敗 → エラーメッセージを表示して中断
   - スキーマ不正 → エラーメッセージを表示して中断
5. システムが確認ダイアログを表示する
   「カテゴリー N 件、リクエスト M 件が含まれています。
    既存のすべてのデータが上書きされます。続けますか？」
6. ユーザーが [キャンセル] → 処理中断、既存データは保持される
7. ユーザーが [OK] → インポート処理を開始する
8. システムが既存の全カテゴリーと全保存済みリクエストを削除する
9. システムがインポートデータを IndexedDB へ書き込む
10. システムが UI を再フェッチして表示を更新する
11. システムが成功フィードバックを表示する
```

### 4.4 エラーハンドリング

| エラー条件 | 表示メッセージ | 対処方法 |
|-----------|--------------|---------|
| ファイルが JSON として不正 | 「ファイルを読み込めませんでした。有効な JSON ファイルを選択してください。」 | `alert()` 表示後中断 |
| `version` フィールドが存在しない / 不正 | 「ファイル形式が正しくありません。api-tester でエクスポートしたファイルを使用してください。」 | `alert()` 表示後中断 |
| `categories` / `requests` が配列でない | 同上 | `alert()` 表示後中断 |
| IndexedDB への書き込みエラー | 「インポート中にエラーが発生しました。もう一度お試しください。」 | `alert()` 表示後中断（部分書き込みの可能性を警告） |
| ファイルサイズが 10 MB 超 | 「ファイルサイズが大きいため、処理に時間がかかる場合があります。」 | `confirm()` で続行確認 |

## 6. データモデル変更

### 6.1 エクスポートファイルスキーマ

```typescript
interface ExportData {
  /** スキーマバージョン（現在は 1） */
  version: 1;
  /** エクスポート日時（Unix タイムスタンプ ms） */
  exportedAt: number;
  categories: Category[];
  requests: SavedRequest[];
}
```

**サンプル JSON**:

```json
{
  "version": 1,
  "exportedAt": 1745798400000,
  "categories": [
    {
      "id": "abc123",
      "name": "Production API",
      "parentId": null,
      "defaultHeaders": [
        { "id": "h1", "key": "Authorization", "value": "Bearer ${TOKEN}", "enabled": true }
      ],
      "defaultParams": [],
      "variables": [
        { "id": "v1", "key": "TOKEN", "value": "secret", "enabled": true }
      ],
      "description": "本番環境",
      "createdAt": 1745798000000
    }
  ],
  "requests": [
    {
      "id": "req1",
      "name": "ユーザー一覧取得",
      "categoryId": "abc123",
      "request": {
        "method": "GET",
        "url": "https://api.example.com/users",
        "params": [],
        "headers": [],
        "body": "",
        "contentType": "application/json"
      },
      "createdAt": 1745798100000
    }
  ]
}
```

### 6.2 IndexedDB スキーマ変更

スキーマ変更なし。既存の `categories`・`saved` オブジェクトストアをそのまま使用する。

### 6.3 lib/storage.ts への追加関数

```typescript
/** 全カテゴリーと全保存済みリクエストをまとめて取得し、エクスポート用データを返す */
export async function exportData(): Promise<ExportData>

/**
 * 既存の全カテゴリーと全保存済みリクエストを削除し、
 * 指定データで置き換える（1 つの IndexedDB トランザクション内で実行）
 */
export async function importData(data: ExportData): Promise<void>
```

`importData()` の実装指針:
- `categories` と `saved` の両ストアを単一トランザクション（`['categories', 'saved']` マルチストアトランザクション）で操作する
- 既存レコードを `store.clear()` で一括削除後、インポートデータを `store.put()` で書き込む
- インポート前にフィールドをストリップし、既知フィールドのみ書き込む

### 6.4 フィールドストリップの基準

インポート時に許容するフィールド:

| 型 | 許容フィールド |
|----|--------------|
| `Category` | `id`, `name`, `parentId`, `defaultHeaders`, `defaultParams`, `variables`, `description`, `createdAt` |
| `SavedRequest` | `id`, `name`, `categoryId`, `request`, `createdAt` |
| `KeyValuePair` | `id`, `key`, `value`, `enabled` |
| `RequestState` | `method`, `url`, `params`, `headers`, `body`, `contentType` |

上記以外のフィールドはインポート時に除外する。

### 6.5 後方互換・マイグレーション方針

- `version: 1` のみサポートする。他のバージョン値は「スキーマ不正」としてエラー扱いにする。
- 将来的に `version: 2` が必要になった場合は、`importData()` 内でバージョン分岐を追加する。

## 7. 受け入れ条件

以下をすべて満たすことでリリース可能とする。

- [ ] **AC-1**: ヘッダーに Export ボタン・Import ボタンが表示されること
- [ ] **AC-2**: Export ボタンをクリックすると `api-tester-export-YYYYMMDD-HHmmss.json` という名前のファイルがダウンロードされること
- [ ] **AC-3**: ダウンロードされた JSON は `{ version: 1, exportedAt: number, categories: [...], requests: [...] }` の構造を持つこと
- [ ] **AC-4**: カテゴリーと保存済みリクエストが 0 件の場合でもエクスポートが成功し、`categories: []`・`requests: []` を含む JSON が生成されること
- [ ] **AC-5**: 正常な JSON ファイルをインポートすると確認ダイアログが表示されること
- [ ] **AC-6**: 確認ダイアログでキャンセルすると既存データが保持されること
- [ ] **AC-7**: 確認ダイアログで OK するとカテゴリーとリクエストが置き換えられ、UI が更新されること
- [ ] **AC-8**: JSON として不正なファイルを選択するとエラーメッセージが表示され、既存データが保持されること
- [ ] **AC-9**: `version` フィールドが存在しない JSON を選択するとエラーメッセージが表示され、既存データが保持されること
- [ ] **AC-10**: エクスポートした JSON ファイルをそのままインポートすると、元の状態が完全に復元されること（往復テスト）
- [ ] **AC-11**: 既存の履歴（`HistoryItem`）はインポート後も保持されること（上書き対象外）
- [ ] **AC-12**: ヘッダーのテーマ切り替えボタン・サイドバーの開閉など既存機能が壊れていないこと（リグレッションなし）

## 8. 備考・制約

- **依存する他機能**: なし（IndexedDB 操作のみ）
- **既存機能への影響**:
  - `ApiTester.tsx` のヘッダー部分にボタンを追加する
  - `lib/storage.ts` に `exportData()` / `importData()` を追加する。既存関数への変更はない
  - `lib/types.ts` に `ExportData` インターフェースを追加する
- **既知の制約**:
  - `window.confirm()` / `window.alert()` をダイアログに使用するため、ブラウザのデフォルト UI が表示される。カスタムモーダルへの差し替えは将来対応とする
  - Safari は `<a download>` によるダウンロードで挙動が異なる場合があるが、Blob URL 方式（`URL.createObjectURL`）で統一することで対応する
- **参考資料**: なし
