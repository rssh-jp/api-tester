# ドキュメント概要

このディレクトリは **api-tester** プロジェクトの仕様書・設計書を管理します。

api-tester は Talend API Tester ライクな REST API テストツールです。  
Next.js + React + TypeScript で構築されており、左ペインにカテゴリーツリー、右ペインにリクエスト編集とレスポンス表示を備えています。カテゴリーは入れ子構造に対応し、ルートカテゴリーのデフォルトヘッダー・パラメータが最高優先度で子カテゴリーに継承されます。

---

## ディレクトリ構成

```
docs/
├── README.md             # このファイル（ドキュメント全体の概要）
├── specs/                # 仕様書（What を定義する）
│   ├── README.md         # 仕様書ディレクトリの説明・命名規則
│   ├── SPEC-TEMPLATE.md  # 仕様書テンプレート
│   └── SPEC-<機能名>.md  # 各機能の仕様書
└── design/               # 設計書（How を定義する）
    ├── README.md         # 設計書ディレクトリの説明・命名規則
    ├── DESIGN-TEMPLATE.md # 設計書テンプレート
    └── DESIGN-<機能名>.md # 各機能の設計書
```

---

## 各ディレクトリの役割

### `specs/` — 仕様書

**「何を作るか」** を定義します。

- 機能要件・非機能要件を記述します。
- プロダクトオーナーや利用者視点で「どう動くべきか」を明文化します。
- UI/UX の動作仕様、API の入出力仕様、受け入れ条件を含みます。
- 実装方法には踏み込まず、**外部から見た振る舞い** にフォーカスします。

### `design/` — 設計書

**「どう作るか」** を定義します。

- 仕様書に対応するコンポーネント設計・型定義・ストレージ設計・API ルート設計を記述します。
- 開発者視点で実装方針を明文化します。
- 実装手順やテスト計画も含みます。
- 原則として、対応する仕様書（`SPEC-<機能名>.md`）が先に存在することを前提とします。

---

## 命名規則

| 種別     | ファイル名               | 例                          |
| -------- | ------------------------ | --------------------------- |
| 仕様書   | `SPEC-<機能名>.md`       | `SPEC-CategoryTree.md`      |
| 設計書   | `DESIGN-<機能名>.md`     | `DESIGN-CategoryTree.md`    |

- `<機能名>` は **PascalCase** または **ハイフン区切り** で記述します。
- 同一機能の仕様書と設計書は同じ `<機能名>` 部分を使用します。
- テンプレートファイル（`SPEC-TEMPLATE.md` / `DESIGN-TEMPLATE.md`）は実際の仕様書・設計書ではなく、新規作成時の雛形です。

---

## ドキュメント作成フロー

```
1. 仕様を決める
   → specs/SPEC-TEMPLATE.md をコピーして specs/SPEC-<機能名>.md を作成

2. 設計を決める
   → design/DESIGN-TEMPLATE.md をコピーして design/DESIGN-<機能名>.md を作成
   → 「対応仕様書」セクションに仕様書ファイル名を記載

3. 実装する
   → 実装完了後、設計書の「実装手順」を必要に応じて更新

4. レビュー・マージ
   → PR に仕様書・設計書も含めてレビューを依頼
```
